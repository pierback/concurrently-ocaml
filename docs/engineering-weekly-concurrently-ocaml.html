Porting npm concurrently to OCaml

A Codex goal-driven port: how we steered the work, why we chose a native runtime, what shipped, and what the benchmarks show.

Target: npm concurrently@10.0.0 May 27, 2026
Keep the opening practical. This is a story about using Codex to push a serious port through compatibility, packaging, and benchmarking.
1. How we did it

We used Codex goals to keep the port strict.

  1. Set one concrete goal: port concurrently to native OCaml with npm-compatible behavior.
  2. Gave Codex hard constraints: no JS runtime fallback, no backward compatibility layer, validate against pinned upstream behavior.
  3. Worked in slices: CLI surface, terminal output, process behavior, npm packaging, compatibility tests, then performance evidence.
  4. After each slice, Codex ran the relevant build, test, compatibility, smoke, or benchmark command before moving on.
The important part: the goal stayed fixed while implementation slices changed.
This replaces the earlier implementation-detail slide. The audience should understand the workflow: goal, constraints, slices, verification.
2. Why OCaml instead of Node?

Because this CLI pays fixed runtime costs very often.

Node is good at the ecosystem part

  • npm install and package-script ergonomics
  • existing user expectations
  • programmatic import surface

OCaml is better for this runtime path

  • native startup avoids V8/bootstrap overhead
  • typed run state makes invalid lifecycle states loud
  • Eio gives direct-style concurrent process supervision
Decision: keep npm as delivery; move execution into a native binary.
Do not make this a language-war slide. The answer is workload fit: short CLI startup plus process supervision.

Explain the shape, not just the numbers. Startup-heavy cases get the largest gain; command-heavy cases still improve but less dramatically.

What to remember.

  1. Codex goal mode helped keep a long-running port pointed at one fixed outcome.
  2. OCaml was chosen to remove Node runtime startup from the execution path.
  3. The result keeps npm ergonomics while running native code.
  4. The benchmarks show the largest wins where this CLI spends avoidable fixed cost.
